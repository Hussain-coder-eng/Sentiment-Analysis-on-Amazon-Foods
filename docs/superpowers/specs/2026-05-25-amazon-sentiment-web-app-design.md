## Problem Statement

Existing project: a static Python script that runs VADER + RoBERTa on a pre-downloaded
Amazon food reviews dataset (500 reviews max) and generates a local Plotly scatter plot.

Goal: Transform it into a live, publicly accessible Next.js web app where a user inputs
an Amazon ASIN and immediately gets dual-model (VADER + RoBERTa) sentiment analysis on
that product's real reviews — for any Amazon product, not just food.

Audience: AI/ML internship recruiters and hiring managers.

## What Makes This Cool

- **Any product**: enter an ASIN → get real review sentiment on that product in seconds
- **Dual-model disagreement viz**: reviews where VADER and RoBERTa diverge are the most
  interesting — and the best interview talking point
- **Always-on demo**: pre-scored Amazon food reviews dataset is always visible on first
  load, with no external calls. Demo never fails.
- **Live URL on resume**: a deployed product, not a notebook

## Scope Boundaries

**Phase 1 (this document):** ASIN input + demo mode. That's it.
**Phase 2 (future):** Amazon URL → ASIN extraction from URL path.
**Deferred:** Product name search. Not in this design.

## Architecture

```
User
  │
  ├── Loads page → demo mode data served from static JSON (no API call)
  │                frontend fires warmup once per session (sessionStorage guard)
  └── Enters ASIN → GET /api/analyze?asin=<ASIN>
  │     (GET with ASIN as query param — validated via searchParams.get('asin'))
  │
  ▼
/api/analyze (GET, Next.js serverless, maxDuration: 60s — set explicitly in route config)
  ├── 1. Validate ASIN (/^[A-Z0-9]{10}$/, normalize to uppercase)
  ├── 2. KV unavailable check → 503 (all subsequent steps need KV; fail-closed)
  ├── 3. Check KV `asin:v1:<ASIN>:scored` (TTL 24h) → cache hit returns immediately
  ├── 4. Check KV `asin:v1:<ASIN>:failed` (TTL varies by failure class) → cached error returns
  ├── 5. Per-IP rate limit: kv.incr(`rate:<IP>:<hourBucket>`) > 5 → 429
  │     (applies to cache-miss path only; cache hits at steps 3-4 bypass rate limit)
  ├── 6. Inflight lock: kv.set(`asin:v1:<ASIN>:inflight`, '1', {nx:true, ex:90})
  │     → if lock exists: return 202 "Analysis in progress"
  ├── 7. Re-check cache inside lock (prevent quota burn from racing requests)
  ├── 8. Monthly circuit breaker: kv.incr(`quota:canopy:YYYY-MM`) > 90 → 503
  │     (only one request per ASIN reaches here; lock prevents concurrent quota burns)
  ├── 9. Canopy API: fetchCanopyReviewsPage(normalizedAsin, 1) → normalize → dedup by dedupKey
  │     └── Normalize: sanitizeHtml(r.body) → text, r.rating (1-5 validated) → rating, id||hash → dedupKey
  ├── 9.5. If hasNextPage AND reviews.length < 5: kv.incr quota → fetchCanopyReviewsPage(normalizedAsin, 2) → merge by dedupKey
  ├── 9.6. Pre-scoring count gate: if reviews.length < 5 → cache 60min failure, return "Not enough reviews"
  │     (check BEFORE scoring to avoid burning HF quota on insufficient data)
  ├── 10. VADER scoring (sync), HF scoring (sequential per review, 100ms gap, 20 review cap)
  ├── 10.5. Compute disagreement per review; build ReviewScore[]
  ├── 11. Post-scoring validation: allScores.length === reviews.length AND all fields finite
  ├── 12. Store result in KV (success: TTL 24h; failure: TTL varies by class)
  ├── 13. Delete inflight lock (finally block — always runs)
  └── 14. Return sanitized JSON

/api/warmup (GET, rate-limited: 1 req/min per IP via KV to prevent HF quota abuse)
  └── await POST to HF API with one dummy string → returns 200 only after HF responds

/api/demo (GET)
  └── Reads public/demo-data.json (raw ReviewScore[]) → wraps and returns:
      { reviews: ReviewScore[], count: number, asin: null }
  └── Same envelope as /api/analyze response — frontend uses data.reviews for both
  └── If file missing at runtime: returns 500 with "Demo data not found" (build-time
      validation in next.config.js prevents deploy without demo-data.json)
```

## Dataset & Demo Data (Step 0 — must do before building UI)

The existing `amazon-data/Reviews.csv` is from Kaggle:
https://www.kaggle.com/datasets/snap/amazon-fine-food-reviews?select=Reviews.csv
(~300MB — do NOT commit to git. Add to .gitignore.)

**Preprocessing script** (run once locally, commit the output):
```
python scripts/preprocess_demo.py
```
This script:
1. Reads Reviews.csv, takes first 500 rows
2. Runs VADER (Python `vaderSentiment`) on each review text
3. Calls HF Inference API sequentially per review (one POST per string, 200ms gap, retry on 503)
   NOTE: array batch input is unvalidated in the Node.js app; Python script uses per-review calls.
   Group into logical chunks of 10 for progress logging (still sequential, 1s pause every 10 reviews).
   Requires: `pip install vaderSentiment requests beautifulsoup4`
   (beautifulsoup4 used for HTML stripping in Python preprocessing script)
4. Computes `disagreement = |vader_compound - (roberta_positive - roberta_negative)|`
5. Writes `public/demo-data.json` using the **shared ReviewScore schema** (same shape as live API response):
   ```json
   [
     {
       "text": "...",
       "rating": 4,
       "vader": { "compound": 0.72, "pos": 0.18, "neg": 0.0, "neu": 0.82 },
       "roberta": { "positive": 0.89, "neutral": 0.08, "negative": 0.03 },
       "disagreement": 0.17
     },
     ...
   ]
   ```
   IMPORTANT: field is `rating` (not `stars`) — matches live-path normalizer output.
   IMPORTANT: `disagreement` is pre-computed here — frontend uses same field for demo and live.
6. Build-time validation in `next.config.js`:
   - Verify `public/demo-data.json` exists, is valid JSON, has ≥100 rows
   - Verify each row has: `text` (string, non-empty), `rating` (number 1-5),
     `vader.compound` (finite, -1 to 1), `vader.pos` (finite, 0-1), `vader.neg` (finite, 0-1),
     `vader.neu` (finite, 0-1), `roberta.positive/neutral/negative` (all finite, 0-1),
     `disagreement` (finite, 0-2)
   - Fail build loudly if any row is malformed — "demo never fails" requires schema integrity
7. Commit `public/demo-data.json` to git (it's ~200KB, not the CSV)

`/api/demo` reads and returns this static file. No external calls.

## VADER Note

Use `vader-sentiment` npm package. It produces the same compound score as Python
`vaderSentiment` for standard review text. Minor differences on emoji-heavy text are
acceptable and documentable in the README as a known limitation.

**VADER output validation** (required — parallel defensive pattern to RoBERTa):
```typescript
function validateVader(v: { compound: number; pos: number; neg: number; neu: number }): VaderScore {
  const fields: (keyof typeof v)[] = ['compound', 'pos', 'neg', 'neu'];
  for (const f of fields) {
    if (!Number.isFinite(v[f])) throw new Error(`vader_invalid_${f}`);
  }
  if (v.compound < -1 || v.compound > 1) throw new Error(`vader_compound_out_of_range_${v.compound}`);
  return v;
}
// Call immediately after vader-sentiment scores each review:
const vaderRaw = vader.SentimentIntensityAnalyzer.polarity_scores(text);
const vaderScore = validateVader(vaderRaw); // throws on malformed output — no silent NaN propagation
```

If an interviewer probes score fidelity: swap `/api/analyze`'s VADER step to call a
Vercel Python runtime function (`/api/vader.py`) — one-file change. Mention this in the
README as "Python VADER available via feature flag."

## Scatter Plot Axes

HF Inference API for `cardiffnlp/twitter-roberta-base-sentiment` returns (verified):
`[{"label":"negative","score":0.01},{"label":"neutral","score":0.09},{"label":"positive","score":0.90}]`
Labels are lowercase strings. All 3 returned by default. Parse into: `{ positive, neutral, negative }` summing to ~1.0.

**Axes:**
- X-axis: VADER compound score (-1 to +1)
- Y-axis: RoBERTa positive score (0 to 1)
- Color: **disagreement score** (blue = agreement, red = high disagreement) — more informative
  than coloring by VADER compound (which duplicates the x-axis encoding)
- Hover: review text (plain text, HTML-stripped) + all 7 scores (4 VADER + 3 RoBERTa)
  NOTE: Amazon review `body` field can contain HTML. Strip it before storing/returning.
  Stripping happens in `/api/analyze/route.ts` (server-side — no DOM available in Node.js).
  Use `sanitize-html` (npm) with `allowedTags: []` to safely extract plain text:
  ```typescript
  import sanitizeHtml from 'sanitize-html';
  // In fetchCanopyReviews normalizer (replace r.body.trim().slice(0, 1000) with):
  const plainText = sanitizeHtml(r.body ?? '', { allowedTags: [], allowedAttributes: {} })
    .trim().slice(0, 1000);
  ```
  Do NOT use regex tag stripping — misses encoded entities and malformed markup.
  Do NOT use `new DOMParser()` in API routes — Node.js has no DOM; crashes at runtime.
  `sanitize-html` handles both entity decoding and malformed tags correctly server-side.

**Disagreement metric** (uncalibrated divergence heuristic — not a calibrated distance):
```
vader_sentiment   = vader_compound                        // -1=neg, +1=pos (native scale)
roberta_sentiment = roberta_positive - roberta_negative   // [-1,+1] range (same direction)
disagreement = |vader_sentiment - roberta_sentiment|      // range: [0, 2]
```
Both operands span [-1, +1] so the difference is directionally comparable, though not
calibrated (VADER compound and RoBERTa difference-score are not trained on the same scale).
Document in README as: "disagreement is a useful heuristic, not a statistically calibrated distance."
Do NOT use `(vader_compound + 1) / 2` normalization — collapses neutral cluster incorrectly.

Disagreement panel: sort descending by `disagreement`, show top 10.

Neutral reviews cluster near (0, ~0.1 on Y-axis). High-disagreement reviews appear in
upper-left (VADER negative, RoBERTa positive) and lower-right (VADER positive, RoBERTa
negative). Do NOT use `(vader_compound + 1) / 2` for normalization — it produces false
disagreement signals for neutral text.

## HF Inference API Call

**Official spec:** `inputs` is a single string. Batch array input is not documented.
Pipe 1 spike must validate whether array input works for this model on HF Inference.
If array input fails: fall back to sequential calls (one POST per review).

**Labels returned (actual):** lowercase strings `negative`, `neutral`, `positive`.
NOT `LABEL_0/LABEL_1/LABEL_2` — those are raw logit indices, not API-level labels.
All 3 labels returned by default (no `top_k` param needed).

```js
// /lib/hf.ts — single-review call (safe; array fallback if spike validates it)
async function scoreOneReview(text: string): Promise<RobertaScore> {
  const res = await fetch(
    'https://api-inference.huggingface.co/models/cardiffnlp/twitter-roberta-base-sentiment',
    {
      method: 'POST',
      headers: {
        Authorization: `Bearer ${process.env.HF_API_KEY}`,
        'Content-Type': 'application/json',
      },
      body: JSON.stringify({ inputs: text }), // single string per spec
      signal: AbortSignal.timeout(8000),
    }
  );
  if (!res.ok) throw new Error(`hf_${res.status}`);
  const labels: { label: string; score: number }[] = await res.json();
  // Response: [{"label":"negative","score":0.01},{"label":"neutral","score":0.09},{"label":"positive","score":0.90}]
  // Validate: all 3 labels present, all scores are finite numbers, sum ≈ 1.0
  const REQUIRED_LABELS = ['negative', 'neutral', 'positive'];
  for (const l of REQUIRED_LABELS) {
    const entry = labels.find(x => x.label === l);
    if (!entry || !Number.isFinite(entry.score)) {
      throw new Error(`hf_label_missing_${l}`); // fail loudly — don't produce silent 0s
    }
  }
  const sum = labels.reduce((s, x) => s + x.score, 0);
  if (sum < 0.95 || sum > 1.05) throw new Error(`hf_score_sum_invalid_${sum.toFixed(3)}`);
  const get = (l: string) => labels.find(x => x.label === l)!.score;
  return {
    positive: get('positive'),
    neutral: get('neutral'),
    negative: get('negative'),
  };
}

// If Pipe 1 confirms array input works, upgrade to batch path:
// body: JSON.stringify({ inputs: reviewTexts }) // array of strings
// Response: [[{label:'negative',...},{label:'neutral',...},{label:'positive',...}], ...]
// One inner array per input string. Labels still lowercase.
```

## HF Cold Start — Mitigation

Vercel function timeout: set `maxDuration = 60` explicitly in route config.
HF cold start: 10-15s. With 20 sequential reviews at ~500ms each = ~10s warm, ~30s cold.
Total worst-case: 45s. Budget accommodated by maxDuration=60.

**Solution: /api/warmup** (rate-limited to prevent HF quota abuse)
```typescript
// Called by the frontend on page load (useEffect, no await)
// sessionStorage guard: only fires once per browser session
if (!sessionStorage.getItem('warmup-done')) {
  fetch('/api/warmup');
  sessionStorage.setItem('warmup-done', '1');
}

// /api/warmup/route.ts — env validation + rate-limited (1 req/min per IP via KV)
// Missing HF_API_KEY: return 503 before any external call (same policy as /api/analyze)
// KV unavailable: fail warmup with 503 — don't proceed unchecked (same as analyze)
export async function GET(request: Request) {
  if (!process.env.HF_API_KEY) {
    return Response.json({ ok: false, error: 'Service misconfigured' }, { status: 503 });
  }
  const ip = request.headers.get('x-real-ip') ?? 'unknown';
  let count: number;
  try {
    const minBucket = Math.floor(Date.now() / 60_000);
    count = await kv.incr(`warmup:${ip}:${minBucket}`);
    if (count === 1) await kv.expire(`warmup:${ip}:${minBucket}`, 120);
  } catch {
    return Response.json({ ok: false }, { status: 503 }); // KV unavailable — fail-closed
  }
  if (count > 1) return Response.json({ ok: false }, { status: 429 });

  let hfRes: Response;
  try {
    hfRes = await fetch('https://api-inference.huggingface.co/models/cardiffnlp/twitter-roberta-base-sentiment', {
      method: 'POST',
      headers: { Authorization: `Bearer ${process.env.HF_API_KEY}`, 'Content-Type': 'application/json' },
      body: JSON.stringify({ inputs: 'warmup' }), // single string per HF spec
      signal: AbortSignal.timeout(15000),
    });
  } catch {
    return Response.json({ ok: false, error: 'hf_timeout' }, { status: 503 });
  }
  if (!hfRes.ok) return Response.json({ ok: false, error: `hf_${hfRes.status}` }, { status: 503 });
  return Response.json({ ok: true });
}
```

Warmup fires on page load. By the time the user enters an ASIN and clicks "Analyze"
(typically 10-30 seconds), the model is warm for sequential scoring.

**First-submit race** (user submits before warmup completes): Show loading spinner
"Waking up the ML model — this takes ~10-30s on first use." The 60s maxDuration
accommodates cold start + full sequential scoring within a single function invocation.

## Canopy API — Review Fetching

Firecrawl replaced with Canopy API (canopyapi.co). Reasons: Firecrawl has 33% success rate
on bot-protected sites and known Amazon failures. Canopy is purpose-built for Amazon,
returns structured JSON with no regex parsing, 100 free requests/month (no credit card).

```typescript
// /api/analyze/route.ts — Canopy review fetch (verified endpoint + field names)
// Endpoint: https://api.canopyapi.co/v1/amazon/product/reviews
// Auth header: API-KEY (not X-Api-Key)
// Review fields: body (text), id, rating, verifiedPurchase (not verified_purchase)
// Pagination: reviewsPaginated.reviews[], hasNextPage, totalResults, totalPages

interface CanopyReviewPage {
  reviews: Review[];
  hasNextPage: boolean;
  totalResults: number;
}

// Single function for all pages — page param replaces the page=1 hardcoded path
// Expects pre-normalized ASIN (uppercase, trimmed) from caller — does NOT re-normalize
async function fetchCanopyReviewsPage(asin: string, page: number): Promise<CanopyReviewPage> {
  const res = await fetch(
    `https://api.canopyapi.co/v1/amazon/product/reviews?asin=${asin}&domain=US&page=${page}`,
    {
      headers: { 'API-KEY': process.env.CANOPY_API_KEY! },
      signal: AbortSignal.timeout(8000), // hard deadline — don't wait past function budget
    }
  );
  if (!res.ok) throw new Error(`canopy_${res.status}`); // 401/403/429/5xx all throw

  const data = await res.json();

  // Actual response shape: data.reviewsPaginated.reviews[] (paginated path)
  // Fallback: data.topReviews[] (top-helpful reviews, non-paginated)
  const raw: any[] = data?.reviewsPaginated?.reviews ?? data?.topReviews ?? [];

  const { createHash } = await import('crypto');

  // Sanitize FIRST, then filter — a body with only HTML tags produces empty text after stripping
  const normalized: Review[] = raw
    .map((r: any) => {
      const id = String(r.id ?? '');
      // sanitize-html strips HTML tags + decodes entities server-side (no DOM needed)
      const text = sanitizeHtml(r.body ?? '', { allowedTags: [], allowedAttributes: {} }).trim().slice(0, 1000);
      const rating = Math.round(Number(r.rating));
      // dedupKey: resolved once here — id if present, else content hash
      // Stored on Review so callers (page-2 cross-page dedup) don't recompute it
      const dedupKey = id || createHash('sha1').update(`${text}_${rating}`).digest('hex').slice(0, 16);
      return { id, dedupKey, text, rating, verified: Boolean(r.verifiedPurchase ?? false) };
    })
    .filter((r: Review) => {
      // Filter on sanitized text (not raw body) to catch tag-only bodies
      return r.text.length > 0 && r.rating >= 1 && r.rating <= 5;
    });

  // Dedup within this page using resolved dedupKey (handles empty-id reviews correctly)
  const seen = new Set<string>();
  const deduped = normalized.filter(r => {
    return seen.has(r.dedupKey) ? false : (seen.add(r.dedupKey), true);
  });

  // Return results + pagination metadata so caller decides whether to fetch page 2.
  // Page 2 quota check MUST happen in the caller (/api/analyze step 9.5), not here.
  // fetchCanopyReviewsPage has no KV context and cannot call kv.incr() for page 2.
  return {
    reviews: deduped,
    hasNextPage: data?.reviewsPaginated?.hasNextPage ?? false,
    totalResults: data?.reviewsPaginated?.totalResults ?? 0,
  };
}
```

In `/api/analyze` (step 9 → 9.5), caller usage:
```typescript
// Step 9: Fetch page 1
const page1Result = await fetchCanopyReviewsPage(normalizedAsin, 1);
let reviews: Review[] = [...page1Result.reviews];

// Step 9.5: Conditionally fetch page 2 if needed
if (page1Result.hasNextPage && reviews.length < 5) {
  // Page 2 costs a second Canopy call — increment quota BEFORE fetching
  const used2 = await kv.incr(quotaKey);
  if (used2 > 90) {
    // Lock released by the outer finally block — no explicit del needed here
    return Response.json({ error: 'Monthly lookup quota reached. Demo mode available.' }, { status: 503 });
  }
  // Use dedupKey (not id) — correctly handles reviews where Canopy omits the id field
  const seenKeys = new Set(reviews.map((r: Review) => r.dedupKey));
  const page2Result = await fetchCanopyReviewsPage(normalizedAsin, 2);
  // Page 2 fetch error (canopy_429/5xx): rethrow — don't silently return page-1-only data
  const newReviews = page2Result.reviews.filter((r: Review) => !seenKeys.has(r.dedupKey));
  reviews.push(...newReviews);
}
// Page 2 fetch error (canopy_429/5xx): rethrow with specific error class,
// NOT silently return page-1-only data — prevents caching transient failures
// as "not enough reviews" for 60min.
```

`fetchCanopyReviewsPage(asin, page)` is the same as `fetchCanopyReviews` with an added `page` query param — extract the shared page-fetching logic into one function taking `(asin: string, page: number): Promise<CanopyReviewPage>`.

If `reviews.length < 5` after page 1 + optional page 2: cache as 60min failure (product-specific), return "Not enough reviews."
Pagination fields confirmed: `reviewsPaginated.hasNextPage` (boolean), `totalResults` (number), `totalPages` (number).

**HF scoring — sequential per review** (official spec: `inputs` is single string):
```typescript
// Sequential calls — safe per official HF spec (inputs = single string).
// Pipe 1 spike validates whether array batch input works. If it does,
// upgrade to chunked batch (10 reviews per POST) to save latency.
// Each call: ~200-500ms warm, ~10s cold. Cap 20 reviews max.
const allScores: RobertaScore[] = [];
for (let i = 0; i < reviews.length; i++) {
  const score = await scoreOneReview(reviews[i].text); // defined in HF section above
  allScores.push(score);
  if (i < reviews.length - 1) await new Promise(r => setTimeout(r, 100)); // 100ms gap
}
if (allScores.length !== reviews.length) throw new Error('hf_length_mismatch');
// Labels: 'negative', 'neutral', 'positive' (lowercase strings, not LABEL_0/1/2)

**If Pipe 1 confirms array batch works**, upgrade to chunked POST:
```typescript
// Batch path (only if spike-validated):
const CHUNK = 10;
for (let i = 0; i < reviews.length; i += CHUNK) {
  const batch = reviews.slice(i, i + CHUNK).map(r => r.text);
  const res = await fetch(HF_URL, { method:'POST', headers:HF_HEADERS,
    body: JSON.stringify({ inputs: batch }), signal: AbortSignal.timeout(8000) });
  if (!res.ok) throw new Error(`hf_${res.status}`);
  // Batch response: [[{label:'negative',...},{label:'neutral',...},{label:'positive',...}], ...]
  const batchScores: { label: string; score: number }[][] = await res.json();
  // Validate response length matches batch length BEFORE iterating — catches partial returns
  if (batchScores.length !== batch.length) throw new Error(`hf_batch_length_mismatch_${batchScores.length}_vs_${batch.length}`);
  for (const labels of batchScores) {
    // Apply same validation as scoreOneReview — no silent zeroes in batch path
    for (const l of ['negative', 'neutral', 'positive']) {
      const entry = labels.find(x => x.label === l);
      if (!entry || !Number.isFinite(entry.score)) throw new Error(`hf_batch_label_missing_${l}`);
    }
    const sum = labels.reduce((s, x) => s + x.score, 0);
    if (sum < 0.95 || sum > 1.05) throw new Error(`hf_batch_score_sum_invalid_${sum.toFixed(3)}`);
    const get = (l: string) => labels.find(x => x.label === l)!.score;
    allScores.push({ positive: get('positive'), neutral: get('neutral'), negative: get('negative') });
  }
  if (i + CHUNK < reviews.length) await new Promise(r => setTimeout(r, 200));
}
// Final length check applies to BOTH sequential and batch paths
if (allScores.length !== reviews.length) throw new Error('hf_final_length_mismatch');

## API Budget

Canopy API free tier: **100 requests/month** (no credit card, permanent).

**Correct `/api/analyze` control flow order** (lock before quota — prevents concurrent quota burns):

```
1.  Validate ASIN (normalize to uppercase before all KV key construction)
2.  KV unavailable check → 503 immediately (all subsequent steps need KV)
3.  Check cache: `asin:v1:<ASIN>:scored` → return hit (no quota burn, bypasses rate limit)
4.  Check failure cache: `asin:v1:<ASIN>:failed` → return cached error (no quota burn)
5.  Per-IP rate limit (kv.incr rate key) — applies to cache-miss path only
6.  Inflight lock: kv.set(`asin:v1:<ASIN>:inflight`, '1', {nx:true, ex:90})
    → if lock exists: return 202 "Analysis in progress, retry in 30s"
    TTL=90s: must exceed max function execution (Canopy 8s + HF 20 reviews×500ms = ~18s + overhead)
7.  Re-check cache inside lock (second concurrent request may have cached result by now)
    → if found: release lock + return cached result
8.  Monthly circuit breaker (kv.incr quota key) ← only reached by ONE request per ASIN
    (lock at step 6 prevents 50 concurrent misses from burning 50 quota slots)
    NOTE: If pagination fetches page 2, that's a second Canopy call = second quota increment.
    Call kv.incr again before page 2 fetch.
9.  Canopy API: `page1Result = fetchCanopyReviewsPage(normalizedAsin, 1)` → normalize → dedup by dedupKey
    Result stored in `let reviews: Review[] = [...page1Result.reviews]`
9.5. If page1Result.hasNextPage AND reviews.length < 5:
    - kv.incr(quotaKey) → if > 90: outer finally releases lock, return 503 quota
    - fetchCanopyReviewsPage(normalizedAsin, 2) → dedup against page-1 seenKeys (dedupKey) → merge
    - If still reviews.length < 5 after page 2: cache as 60min failure, return "Not enough reviews"
9.6. Pre-scoring count check: if reviews.length < 5: cache 60min failure, return "Not enough reviews"
    (Doing this check BEFORE scoring avoids burning HF quota on products that fail the gate)
10. VADER sync scoring + HF sequential scoring (one POST per review, 100ms gap)
10.5. Compute `disagreement` per review (attach to scored object before storing):
    ```typescript
    const scored: ReviewScore[] = reviews.map((r, i) => {
      const vader = vaderScores[i];
      const roberta = robertaScores[i];
      const disagreement = Math.abs(vader.compound - (roberta.positive - roberta.negative));
      return { text: r.text, rating: r.rating, vader, roberta, disagreement };
    });
    ```
11. Quality gate: scored.length >= 5 AND every item has all required fields
12. Store result (success: TTL 24h; failure TTL by class — see Error States table)
13. Delete inflight lock (finally block — executes even if steps 9-12 throw)
14. Return sanitized JSON — response schema matches demo-data.json exactly:
    ```json
    {
      "reviews": [
        {
          "text": "plain text, HTML-stripped",
          "rating": 4,
          "vader": { "compound": 0.72, "pos": 0.18, "neg": 0.0, "neu": 0.82 },
          "roberta": { "positive": 0.89, "neutral": 0.08, "negative": 0.03 },
          "disagreement": 0.17
        }
      ],
      "count": 15,
      "asin": "B07FZ8S74R"
    }
    ```
    Frontend uses `data.reviews` for both demo and live — same accessor, same `ReviewScore[]` type.
    `/api/demo` returns `{ reviews: ReviewScore[], count: number, asin: null }` (not a raw array)
    to match this envelope shape exactly. No mapping layer needed.
```

**Monthly circuit breaker** (KV-enforced — incremented at step 8 for page 1; incremented again at step 9.5 for each additional Canopy page fetch):
```typescript
const monthBucket = new Date().toISOString().slice(0,7); // '2026-05'
const quotaKey = `quota:canopy:${monthBucket}`;
const used = await kv.incr(quotaKey);
if (used === 1) await kv.expire(quotaKey, 60 * 60 * 24 * 32);
if (used > 90) {
  // Lock released by the outer finally block — no explicit del needed here
  return Response.json(
    { error: 'Monthly lookup quota reached. Demo mode available.' },
    { status: 503 }
  );
}
```

**Inflight lock** (atomic, TTL 90s — step 6):
```typescript
const normalizedAsin = asin.trim().toUpperCase();
// SET NX EX — atomic: returns null if key already exists
const locked = await kv.set(`asin:v1:${normalizedAsin}:inflight`, '1', { nx: true, ex: 90 });
if (!locked) return Response.json(
  { error: 'Analysis already in progress for this product. Retry in 30s.' },
  { status: 202 }
);
try {
  // ... steps 7-13 ...
} finally {
  // Non-throwing cleanup — lock release failure must not mask the real error or response
  try { await kv.del(`asin:v1:${normalizedAsin}:inflight`); } catch (e) { console.error('lock_release_failed', e); }
}
```

At 90/100 requests, live lookup disables. Pre-populate 5-10 notable ASINs at deploy time.

## Rate Limiting

Per-IP rate limit on `/api/analyze` (cache-miss path only) to protect Canopy API quota:

```typescript
// Vercel-specific: x-real-ip is set by Vercel's edge and cannot be spoofed
// by clients when deployed on Vercel. This assumption is ONLY valid on Vercel —
// document this explicitly. On other hosts, x-real-ip may be forged.
// IMPORTANT: This header behavior is Vercel-specific. Do not use without this guard
// in non-Vercel deployments.
const ip = (request.headers.get('x-real-ip') ??
             request.headers.get('x-forwarded-for')?.split(',')[0]?.trim() ??
             'unknown'); // 'unknown' collapses header-absent clients — document as known limitation
const hourBucket = Math.floor(Date.now() / 3_600_000);
const key = `rate:${ip}:${hourBucket}`;
const count = await kv.incr(key);
if (count === 1) await kv.expire(key, 3600);
if (count > 5) return Response.json(
  { error: 'Too many requests. Please wait before trying another product.' },
  { status: 429 }
);
```

**Rate limit applies to cache-miss path only** (steps 3-4 return before rate limit check).
Cache hits do NOT count toward rate limit — they consume no Canopy quota.

**Double-click behavior** (intended): A rapid double-click sends two near-simultaneous cache-miss
requests for the same ASIN. Both pass rate limit (each increments the hourly counter by 1).
The first acquires the inflight lock (step 6); the second sees the lock and returns 202
"Analysis in progress." This is the correct UX: the user sees the "in progress" message
rather than two parallel Canopy calls. Frontend should disable the submit button after first click.

**Global safety net:** The monthly circuit breaker (90 req cap) is the backstop against
IP-rotating attackers who cycle through VPNs to bypass per-IP limits. A VPN list can burn
the 90-request cap, but cannot exceed it. This is an acceptable tradeoff for a portfolio project;
a production system would add CAPTCHA or signed client nonces.

KV unavailable during rate limit check → **return 503, disable live lookup** (fail-closed).

## Error & Failure States

Failure cache TTL varies by class — transient errors use short TTL (retry soon);
product-specific failures use longer TTL (won't change until new reviews posted).

| Failure | User sees | Failure cache TTL |
|---------|-----------|-------------------|
| Invalid ASIN format | "Invalid ASIN. Must be 10 alphanumeric characters (e.g. B07XXXXX1X)." | No cache (client-side validation) |
| Valid ASIN, 0 reviews | "No reviews found for this product. Try another ASIN or use demo mode." | 60min (product won't gain reviews soon) |
| Canopy 429 (rate limited) | "Couldn't fetch reviews right now. Try again in a few minutes." | 5min (transient — Canopy quota resets) |
| Canopy 4xx (auth/bad req) | "Couldn't fetch reviews. Try demo mode or a different product." | 30min (likely config issue, not transient) |
| Canopy 5xx / timeout | "Couldn't fetch reviews. Try again in a few minutes." | 2min (transient server error) |
| Canopy returns < 5 reviews | "Not enough reviews to analyze. Try a more popular product." | 60min (product-specific) |
| HF API cold start | Loading spinner: "Waking up the ML model — ~10s on first use." | No cache (UI state) |
| HF API non-200 / timeout | "Model taking too long. Try again in 30 seconds." | 2min (transient) |
| HF length mismatch | "Scoring failed. Try again in a moment." | 1min (internal bug — short TTL to allow retry) |
| Monthly quota reached (≥90) | "Monthly lookup quota reached. Demo mode available." | No cache (quota state in KV) |
| Rate limit exceeded | "Too many requests. Please wait before trying another product." | No cache (KV tracks directly) |
| KV unavailable | **503: "Service temporarily unavailable. Demo mode still works."** (fail-closed) | No cache (KV is down) |
| Missing env vars at startup | **503** "Service misconfigured" — no provider calls made | No cache |
